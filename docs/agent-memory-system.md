# Agent 记忆系统：从认知科学到工程实现

## 为什么 Agent 需要记忆

LLM 本身是无状态的。每次 API 调用都是独立的计算，模型不会记住上一次对话。这带来两个根本问题：

- **对话遗忘**：用户说"我叫李明"，下一轮问"你记得我叫什么吗"，模型不知道
- **知识局限**：模型只知道训练数据截止前的内容，不知道你系统里的实时业务数据

记忆系统就是解决这两个问题的工程方案。

## 人类记忆的启发

认知科学把人类记忆分成几个层级：

```
感觉记忆（0.5~3秒）→ 工作记忆（15~30秒，7±2 个项目）→ 长期记忆（可达终生）
                                                              ├── 程序性记忆（骑自行车）
                                                              └── 陈述性记忆
                                                                   ├── 语义记忆（巴黎是法国首都）
                                                                   └── 情景记忆（昨天的会议内容）
```

映射到 Agent 系统，我们设计四种记忆类型：

| 人类记忆 | Agent 记忆 | 存什么 | 存多久 |
|---------|-----------|--------|--------|
| 工作记忆 | Working Memory | 当前对话上下文 | 会话结束清除 |
| 情景记忆 | Episodic Memory | 具体事件（"3月15号和李明通了电话"） | 长期保留 |
| 语义记忆 | Semantic Memory | 抽象知识（"李明是保守型客户"） | 永久 |
| 感觉记忆 | Perceptual Memory | 图片、音频等多模态数据 | 按需 |

## 记忆的生命周期

一条信息从产生到消亡，经历五个阶段：

```
编码 → 存储 → 检索 → 整合 → 遗忘
```

- **编码**：把原始对话转成可存储的形式（提取关键信息 + 向量化）
- **存储**：写入对应的记忆层
- **检索**：根据当前问题从记忆里找相关条目
- **整合**：重要的短期记忆"晋升"为长期记忆
- **遗忘**：清理不需要的记忆，释放空间

## 四种记忆的实现细节

### 工作记忆（Working Memory）

最简单的一层，存当前对话的即时上下文。

```
存储方式：纯内存（Python list / Redis）
容量上限：50 条或 N 轮对话
过期机制：TTL 60 分钟，会话结束清除
检索方式：TF-IDF + 关键词混合打分
```

评分公式：

```
最终得分 = (向量相似度 × 0.7 + 关键词匹配 × 0.3) × 时间衰减 × (0.8 + 重要性 × 0.4)
```

工作记忆的作用是让 LLM 能理解指代——用户说"换个激进点的"，工作记忆里有上一轮的推荐结果，LLM 知道"换"是指什么。

### 情景记忆（Episodic Memory）

存具体事件，带时间戳和上下文。

```
存储方式：SQLite（结构化查询） + 向量数据库（语义检索）
写入时机：每轮对话结束后，LLM 异步判断是否有值得记住的事件
数据结构：
  - episode_id: 唯一标识
  - timestamp: 事件时间
  - content: 事件描述（"和李明电话沟通，讨论了华夏成长混合的认购意向"）
  - entities: 涉及的实体（["李明", "华夏成长混合"]）
  - importance: 重要性分数
  - session_id: 所属会话
```

**为什么用两个数据库？**

SQLite 擅长精确条件过滤（`WHERE timestamp > '上周' AND entity = '李明'`），向量库擅长语义匹配（"之前聊过调仓的事"）。检索时先用 SQLite 缩小候选范围，再在候选集内做向量排序。

评分公式：

```
最终得分 = (向量相似度 × 0.8 + 时间近因性 × 0.2) × (0.8 + 重要性 × 0.4)
```

时间近因性权重高——越近的事件越可能是用户关心的。

**什么算"值得记住的事件"：**

| 记 | 不记 |
|---|------|
| 客户状态变化（退休、换工作） | 闲聊寒暄 |
| 明确的投资意向 | 重复已知信息 |
| 约定的跟进时间 | 模糊的询问 |
| 客户反馈（满意/不满） | AI 的通用建议 |

### 语义记忆（Semantic Memory）

存从事件中提炼出的抽象知识，是记忆系统中最复杂的一层。

```
存储方式：向量数据库（语义检索） + 图数据库（关系推理）
写入时机：定期从情景记忆中归纳，或同一模式出现多次时触发
数据结构：
  向量库：知识文本的 embedding（"李明是保守型客户，偏好债券类产品"）
  图库：实体节点 + 关系边（李明 --风险偏好--> 保守型 --适合--> 债券型）
```

**向量库和图库各管什么？**

| 场景 | 向量库 | 图库 |
|------|--------|------|
| "和李明相关的知识" | ✅ 语义匹配 | ✅ 按实体查 |
| "稳健一点的产品" | ✅ 模糊匹配"稳健" | ❌ 没有对应的边 |
| "李明持有的基金的经理是谁" | ❌ 多跳关系查不到 | ✅ 李明→持有→基金→经理 |

两个库存的不是同样的数据：

- 向量库存**完整的知识文本**，向量化后按语义相似度检索
- 图库存**实体和关系**（节点 + 边），按图遍历做链式推理

写入时同一条语义记忆两边都写：文本向量化存 Qdrant，提取实体关系存 Neo4j。检索时两路并行查、加权合并：

```
最终得分 = (向量相似度 × 0.7 + 图检索相似度 × 0.3) × (0.8 + 重要性 × 0.4)
```

### 感知记忆（Perceptual Memory）

处理多模态数据（图片、音频、视频）。用 CLIP 处理图片、CLAP 处理音频，向量化后按模态分别存储。

大部分文本对话场景不需要这一层，跳过。

## 记忆整合：信息如何在层之间流动

三层之间的晋升链路：

```
工作记忆 ──(每轮对话后, importance > 0.7)──→ 情景记忆
情景记忆 ──(定期归纳, 同一模式出现多次)──→ 语义记忆
```

完整示例：

```
第1轮对话："李明想买债券基金"
  → 存入工作记忆
  → 这轮结束，LLM 判断 importance=0.8 > 0.7，晋升为情景记忆

第2轮对话："李明拒绝了股票基金推荐"
  → 存入工作记忆 → 晋升为情景记忆

第3轮对话："李明说只要低风险的"
  → 存入工作记忆 → 晋升为情景记忆

... 定期任务扫描，发现李明相关事件都指向"保守" ...
  → 归纳为语义记忆："李明是保守型客户，只接受低风险产品"
  → 写入向量库 + 图库（李明 --风险偏好--> 保守型）
```

**晋升触发条件：**

| 晋升方向 | 触发条件 | 频率 |
|---------|---------|------|
| 工作 → 情景 | 每轮对话后，importance > 阈值 | 实时（异步） |
| 情景 → 语义 | 同一实体事件累积 N 条 / 定期任务 | 每天或按量触发 |

## 遗忘机制：控制记忆膨胀

三种策略，可组合使用：

| 策略 | 逻辑 | 适用场景 |
|------|------|---------|
| 重要性遗忘 | importance < 阈值 → 删除 | 清理无意义的闲聊记录 |
| 时间遗忘 | 超过 N 天没被访问 → 删除 | 清理过时信息 |
| 容量遗忘 | 存储满了 → 删最不重要的 | 控制存储成本 |

生产环境的遗忘通常不是硬删：

1. 先标记为"待遗忘"
2. 一段时间内没被检索命中 → 真正删除
3. 如果又被命中了 → 恢复，说明还有用

模拟人脑的"越回忆越深刻，不回忆就淡忘"。

## 检索时：三层并行查询

用户提问时，不是逐层查找，而是三层同时检索、结果合并：

```
用户提问："给李明推荐什么基金"
    │
    ├─ 工作记忆："刚才李明说他退休了"（当前上下文）
    │
    ├─ 情景记忆："5月31号李明表达了买债券基金的意向"（历史细节）
    │
    └─ 语义记忆："李明=保守型，适合债券/货币基金"（抽象结论）
    │
    ▼
合并去重，按相关性排序，取 top N
    │
    ▼
拼入 prompt 给 LLM 生成回答
```

三层各有优势：
- 工作记忆：提供**当前上下文**（"刚才说了什么"）
- 情景记忆：提供**历史细节**（"具体哪天聊了什么"）
- 语义记忆：提供**抽象结论**（"这个客户本质上是什么偏好"）

## 上下文管理：怎么把记忆塞进有限的 Context Window

记忆系统解决了"记什么"的问题，上下文管理解决的是"每次调 LLM 时带多少进去"。LLM 的 context window 有上限（8k/32k/128k tokens），不可能把所有记忆都塞进去。

### 五种常见策略

**1. 滑动窗口**

最简单。只保留最近 N 轮对话，更早的直接丢弃。

```
保留最近 10 轮 → 第 11 轮进来时，第 1 轮被丢弃
```

优点：实现简单，零成本。
缺点：按轮数算不精确——用户说"好的"是一轮，贴一大段分析也是一轮。可能 5 轮就超了 token 限制，也可能 20 轮都没满。

**2. 摘要压缩**

超出窗口的部分让 LLM 压缩成一句话摘要，放在 messages 最前面。

```
第 1-5 轮 → LLM 总结为："用户李明想从股票基金转向债券基金，已推荐了中欧纯债A"
第 6-10 轮 → 保留原文
```

优点：保留了关键信息，不完全丢失上下文。
缺点：每次压缩要多调一次 LLM；摘要会丢失细节（语气、具体措辞）。

**3. Token 预算制**

不按轮数算，按实际 token 数精确管理。给 context window 的每个部分分配预算：

```
总 context window = 8000 tokens
├── system prompt:    500 tokens（固定）
├── 记忆检索结果:    2000 tokens（三层记忆合并后的 top N）
├── RAG context:     1500 tokens（知识库召回的 chunks）
├── 当前用户消息:     200 tokens（动态）
├── 预留生成空间:    1000 tokens（给模型回答用）
└── 历史对话:        2800 tokens（从最新往前装，装满为止）
```

从最新消息往前逐条累加 token 数，超出预算的部分截断或压缩。

```python
budget = 2800
used = 0
kept = []
for msg in reversed(history):
    msg_tokens = count_tokens(msg)
    if used + msg_tokens > budget:
        break  # 装不下了
    kept.append(msg)
    used += msg_tokens
# kept 就是最终带进 prompt 的历史消息
```

优点：精确利用 context window，不浪费也不溢出。
缺点：需要一个 token 计数器（tiktoken 等），实现稍复杂。

**和摘要压缩的区别**：核心不在于"超出后怎么处理"（都可以压缩），而在于"怎么衡量超出"——按轮数是粗略近似，按 token 是精确计算。生产环境基本都用 token 预算制。

**4. 递归摘要**

每满 N 轮压缩一次，摘要套摘要，形成层次结构：

```
第 1-5 轮   → 摘要 A
第 6-10 轮  → 摘要 B
第 11-15 轮 → 原文保留

最终 prompt = [摘要A] + [摘要B] + [第11-15轮原文]
```

信息密度逐层递减，但永远不会完全丢失最早的上下文。适合超长会话场景。

**5. 分层记忆（最完整的方案）**

就是本文讲的整套体系。不只是管理对话原文，而是把信息提炼到不同层级：

```
工作记忆：最近几轮原文（在 context window 里）
情景记忆：提取的事件（按需检索，命中才带入）
语义记忆：抽象知识（按需检索，命中才带入）
```

大部分信息不在 context window 里占空间，只有被检索命中时才临时拉进来。相当于把"记忆"从 prompt 里搬到了外部数据库，按需加载。

### 生产环境怎么选

| 场景 | 推荐方案 |
|------|---------|
| 客服对话（单次短交互，3-5 轮） | 滑动窗口就够 |
| 销售助手（多次跟进同一客户） | Token 预算制 + 分层记忆 |
| 代码助手（长 session，几十轮） | Token 预算制 + 递归摘要 |
| 通用聊天（ChatGPT 类） | Token 预算制 + 摘要压缩 |

### 记忆 vs 历史对话

容易混淆的一点：上下文管理管的是"对话历史原文"，记忆系统管的是"从对话中提取的结构化信息"。两者都会出现在最终 prompt 里，但来源和管理方式不同：

```
最终 prompt 组成：
├── system prompt
├── 语义记忆检索结果（"李明是保守型客户"）     ← 记忆系统提供
├── 情景记忆检索结果（"上周和李明聊了债券基金"） ← 记忆系统提供
├── 旧对话摘要（"之前讨论了调仓方案"）          ← 上下文管理压缩的
├── 最近 N 轮对话原文                           ← 上下文管理保留的
└── 当前用户消息
```

## 完整生命周期总结

```
对话产生 → 存入工作记忆 → 晋升情景记忆 → 归纳为语义记忆
              ↕                ↕               ↕
           TTL 过期          时间遗忘        永久保留
           容量淘汰          重要性遗忘      （除非显式删除）
```

## 生产环境选型建议

| 组件 | 推荐方案 | 备选 |
|------|---------|------|
| 工作记忆存储 | Redis（支持 TTL） | 内存 dict |
| 情景记忆结构化 | PostgreSQL | SQLite |
| 情景记忆向量 | Qdrant / Pinecone | ChromaDB |
| 语义记忆向量 | Qdrant / Pinecone | Milvus |
| 语义记忆图 | Neo4j | 不用图库，只用向量 |
| Embedding 模型 | bge-base-zh-v1.5 | text-embedding-3-small |

## 和纯 RAG 的区别

RAG 解决的是"从外部文档里找答案"，记忆系统解决的是"记住交互过程中产生的信息"。两者互补：

| | RAG | 记忆系统 |
|--|-----|---------|
| 数据来源 | 预先灌入的文档 | 对话过程中动态产生 |
| 更新频率 | 低（重新索引） | 高（每轮对话都可能写入） |
| 检索目标 | 找到相关文档片段 | 找到相关的历史交互和知识 |
| 典型场景 | "这个产品的费率是多少" | "上次和李明聊了什么" |
